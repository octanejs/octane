# dbmon bench — octane (TSRX vs JSX) vs react

A fourth benchmark adjacent to [`js-framework`](../js-framework/),
[`recursive-context`](../recursive-context/) and
[`signal-favoring`](../signal-favoring/). Where js-framework measures bulk
create/clear of a wide flat list and recursive-context measures deep Context
fan-out, **dbmon stresses the UPDATE path**: a [dbmonster][dbmon]-style table
whose every cell's value (text) and threshold class churn on every frame.

This is the workload the compiler's per-binding mount/update codegen drives
hardest — so it's the measurement vehicle for ongoing compiler work on
text/class diffing and mount-vs-update normalization.

## Layout

```
benchmarks/dbmon/
├── octane-tsrx/   # Vite app, dev :5196 — octane authored in .tsrx
├── octane-jsx/    # Vite app, dev :5197 — same app authored in React-style .tsx
├── react/         # Vite app, dev :5198 (React 19, production mode)
├── run.mjs        # Playwright harness — drives all adapters
├── package.json   # umbrella: `pnpm bench`
└── README.md
```

> Solid and Ripple columns are intentionally deferred: their fine-grained
> reactive models update dbmon idiomatically through stores / tracked rows
> rather than the immutable-newArray + keyed-reconcile model the three apps
> here share, so they belong in a follow-up that wires each up fairly.

The octane app is authored twice over the same octane core — `.tsrx` (directive
`@for`, `class`) and React-style `.tsx` (`{data.map(... key=)}`, `className`).
Both compile through `octane/compiler/vite`, so the two octane columns read the
JSX backwards-compat path's cost on this update-heavy workload. React is the
canonical VDOM baseline and shares the exact same dataset + keyed-list model.

## Shape

`DB_COUNT = 100` rows (see `src/data.js`), each a `<tr>` of 7 `<td>`s: a name
cell, a query-count cell, and 5 query cells. Every non-name cell carries a value
**and** a threshold class, so a full tick churns 6 cells/row × 100 = 600 cell
diffs. Data is generated by a seeded PRNG keyed on a frame counter, so every
framework renders identical data each frame — an apples-to-apples comparison.

## Ops

| op             | what it stresses                                                        |
| -------------- | ---------------------------------------------------------------------- |
| `mount`        | build the whole table (rows + cells) from empty                        |
| `tick`         | full update — all rows reconcile in place; every cell's text+class diffs |
| `tick_partial` | only ~10% of rows change; rest are value-identical (diff-skip path)     |
| `remount`      | every key is new — bulk unmount of old rows + mount of a fresh set      |
| `sort`         | toggle order — worst-case keyed reorder (LIS moves)                     |
| `unmount`      | tear the whole table down                                              |

All ops flush synchronously, so the harness times only framework JS work (GC
forced before each sample). The harness also prints a `tick_partial / tick`
diff-skip ratio — low means the per-binding diff is skipping unchanged cells.

## Running

Start the three preview servers (production builds), then run the harness:

```bash
# build + preview each (production); run from the repo root
pnpm --filter octane-tsrx-dbmon-bench build && pnpm --filter octane-tsrx-dbmon-bench preview &
pnpm --filter octane-jsx-dbmon-bench  build && pnpm --filter octane-jsx-dbmon-bench  preview &
pnpm --filter react-dbmon-bench       build && pnpm --filter react-dbmon-bench       preview &

# then, from benchmarks/dbmon:
pnpm bench           # 30 timed iterations (+10 warmup) per op
pnpm bench:long      # 50 iterations
```

Swap `build && … preview` for `dev` to measure the unminified dev build. Set
`TARGETS='[{"name":"octane-tsrx","url":"http://localhost:5196/"}]'` to run a
single adapter.
