Coverage observations

- All cited upstream line numbers come from the fixture comments themselves (3 in
  react-conformance.tsrx); the rest of the mappings are by behavior, since the
  other fixtures do not cite line numbers. Lines in
  `ReactHooksWithNoopRenderer-test.js` drift across React versions, so the cited
  lines (311, 1800, 1885, 1926, 3092, 3124) are approximate against React main and
  should be treated as anchor references, not exact pins.
- `hooks.test.ts > useCallback > renders without errors` is a smoke test only;
  identity stability for useCallback is actually covered in `callbacks.test.ts`
  (two stronger tests). The hooks.test.ts case adds little signal versus React's
  upstream battery.
- `effect-timing.test.ts > all phases fire cleanup on unmount` asserts cleanups
  fire in registration order (ins → lay → eff). React's unmount path typically
  runs cleanups in reverse mount order within a fiber; the across-phase ordering
  octane asserts is plausible but is the place most likely to drift from
  React's behavior — flagging Medium confidence.
- No coverage for: `useImperativeHandle` (ReactUseImperativeHandle-test.js),
  `useId` (ReactUseId-test.js), `useDeferredValue`, `useTransition`,
  `useSyncExternalStore`, `useContext`, effect-event-inside-render guards, or
  strict-mode double-invoke. If the goal is React conformance these are the
  obvious next-row additions.
- No coverage for the "effect created during a cleanup" / re-entrant scheduling
  cases that ReactEffectsSemantics-test.js covers.
- `useReducer` lazy init (third arg to useReducer) is not covered; only the basic
  dispatch path is.
