# Differences from React

Octane runs on React's programming model — the same hooks, `memo`, context,
portals, Suspense, transitions, actions, and SSR. We check this the boring way:
over 2,200 conformance tests lifted straight from `facebook/react`. So the
starting assumption is that your React knowledge just works.

The handful of places Octane diverges are listed below, and they're all on
purpose. If Octane behaves differently from React and it isn't on this page,
that's a bug — please file it.

A quick note before the list: most of these won't come up in day-to-day code.
Some are freedoms you'll be glad to have, a couple are things to know if you hit
them, and a few are internal details we've written down for completeness that
you'll never actually feel. We've said which is which.

## Hooks can go anywhere

**This one you'll notice, and you'll like it.** There are no rules of hooks in
Octane. A hook can sit inside an `if` or after an early return — the compiler
tracks each hook by its position in the source, not by the order it runs, so
the usual React footguns just aren't there.

```tsrx
function Profile(props) {
	if (props.loading) return <Spinner />;
	// Totally fine — React would throw here.
	const [name, setName] = useState(props.name);
	@{
		<input value={name} onInput={(e) => setName(e.target.value)} />
	};
}
```

The exception is a plain JS loop. One call site means one hook slot, so every
iteration of a loop would share the same state — instead of letting that fail
silently, the compiler rejects it at build time. Loop in the template with the
keyed `@for` directive (each item renders in its own scope, so per-item hooks
get per-item state), or extract the loop body into a child component. `use()`
and `useContext` do work inside loops — they aren't slot-keyed.

## State hooks have a current-state getter

**This replaces a common React ref workaround.** `useState` and `useReducer`
have an optional third tuple member: a stable thunk that reads the latest state
cell.

```tsrx
const [draft, setDraft, getDraft] = useState('');

const saveLater = async () => {
	await waitForConnection();
	await save(getDraft());
};
```

In React, a callback like this usually needs a ref synchronized on every render
to avoid capturing an old `draft`. In Octane, `getDraft()` reads the latest
scheduled value directly without subscribing or causing a render. During
pending work that value can be newer than the currently committed DOM.

The compiler only emits the getter-enabled hook when tuple index 2 can be
observed. Ordinary `[state, setState]` and `[state, dispatch]` destructures keep
the existing two-item runtime path and allocate no getter; escaped or ambiguous
tuples conservatively receive the full three-item shape.

## Native event objects, no synthetic event layer

Event propagation itself matches React and is **no longer a difference**.
Ordinary bubbling and capture, `stopPropagation`, logical propagation through
portals, and native non-bubbling events such as `toggle`, dialog
`cancel`/`close`, media events, and `load`/`error` all reach the same logical
ancestors React does.

What remains different is the event API and synthesis layer:

- `onClick`, `onInput`, `onSubmit`, and friends receive the browser's actual
  `Event` object rather than a React `SyntheticEvent` wrapper.
- `mouseEnter`/`pointerEnter` are the browser's own per-element events, not
  events synthesized from `mouseover`/`mouseout`.
- There's no synthetic `onChange`, `onBeforeInput`, or `onSelect` polyfill. Use
  the corresponding native event — `onInput` for per-keystroke text changes.

There is no event pooling, and `event.currentTarget` is the handler's element.

## Controlled components, native events

Controlled inputs work exactly like React's. Put `value` or `checked` on an
`<input>`, `<textarea>`, or `<select>` and the prop drives the DOM: Octane
re-asserts it on every render and after events, so a rejected edit snaps
straight back, radio groups stay in sync, and `<select value>` selects the
matching option. IME composition is respected, and `defaultValue` /
`defaultChecked` give you uncontrolled inputs, same as React.

The one thing to rename: there's no synthetic `onChange`. The per-keystroke
event is the native one, `onInput` — the native `change` event fires on blur.
Migration is a one-word diff:

```tsrx
// React
<input value={text} onChange={(e) => setText(e.target.value)} />
// Octane
<input value={text} onInput={(e) => setText(e.target.value)} />
```

If you render a controlled text input without an `onInput` handler, Octane
warns in development — with a specific pointer when you passed `onChange`,
since that fires on blur here, not per keystroke.

Form **actions** are the modern React 19 story and match it:
`<form action={fn}>`, `useActionState`, `useFormStatus`, `useOptimistic`, and
form reset all work as you'd expect.

## `className` is smarter about arrays and objects

Octane composes `class`/`className` clsx-style. Strings, numbers, arrays,
objects, and nested combinations all fold into a class string, and falsy values
drop out:

```tsrx
<div className={['card', isActive && 'active', { selected }]} />
```

A plain string is still the fast path and behaves identically to React. The one
visible difference: React turns `className={['a', 'b']}` into `"a,b"`, while
Octane gives you `"a b"` — which is almost certainly what you meant.

## Errors use `@try`/`@catch`

Octane has no class components, so error boundaries aren't a class lifecycle.
You wrap the risky part in a `@try`/`@catch` block (or the `<ErrorBoundary>`
component) instead:

```tsrx
@try {
	<RiskyThing />
} @catch (err, reset) {
	<div>
		Something broke.
		<button onClick={reset}>Try again</button>
	</div>
};
```

It catches what React's boundaries catch — render errors, and errors thrown
during teardown. The only real difference is that an error nothing catches goes
to `console.error` rather than React's `onUncaughtError` callback.

## SSR gives you the CSS too

Server rendering works the same way, including streaming out-of-order Suspense
content. The one addition worth knowing: `renderToString` and friends return
`{ html, css }`, where `css` carries Octane's scoped styles. React has no
equivalent because it has no built-in scoped styling.

If server and client HTML disagree during hydration, Octane fixes it up in
place — it patches to the client's value and warns in the console — instead of
throwing away the tree.

## Things that differ under the hood (you won't feel these)

For completeness, a couple of internal behaviors differ from React. You can
skip this section unless you're the kind of person who reads runtime source for
fun — none of it changes what your components do.

- **Scheduling is synchronous.** Octane batches renders in a microtask and runs
  them to completion. There are no lanes, time-slicing, or selective hydration.
  `flushSync` drains everything; passive effects always run after paint. The
  user-visible behavior of transitions — keeping the old UI on screen while new
  content suspends — is fully there.
- **The keyed list reconciler uses a different algorithm** to work out the
  minimal set of DOM moves. The final DOM, the identity of the nodes that
  survive, and their order are all identical to React's — only the internal set
  of nodes it physically moves can differ, which nothing observable depends on.

## Not implemented, on purpose

A few React features don't exist in Octane by design:

- **Class components** — Octane is functions only.
- **Server Components / RSC.**
- **`StrictMode` double-invoke, `Profiler`, `SuspenseList`.**
- **`forwardRef` and `createRef`** — refs are just props
  (`ref={cb}`, `ref={obj}`, or `ref={[a, b]}`), so you never need to forward
  them.

Resource hints (`preload`, `preinit`, `preconnect`, `prefetchDNS`) are
supported.
