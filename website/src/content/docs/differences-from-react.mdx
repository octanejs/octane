<h2 id="hooks">Hooks fit the code, not a call order</h2>

React's state and effect model still applies. What changes is where you may call a
hook and how much bookkeeping you need to write.

<h3>Conditional hooks are valid</h3>

Octane assigns each hook call site a stable slot at compile time. A hook can sit
behind a condition or after an early return without shifting the state of another
hook.

```tsrx
export function Profile(props) {
	if (!props.user) return <EmptyState />;

	const [editing, setEditing] = useState(false);
	return <button onClick={() => setEditing(!editing)}>Edit profile</button>;
}
```

The exception is a slot-based hook inside a plain JavaScript `for` or `while` loop.
Every iteration would share one slot, so the compiler reports an error. Use keyed
`@for` or move the hook into a child component. `use()` and `useContext` are exempt
because they are not slot-keyed.

<h3>Dependency lists are optional</h3>

Leave the dependency list off an effect or memo and the compiler reads the values
captured by its callback:

```tsrx
useEffect(() => {
	syncRoom(room.id);
}); // inferred from syncRoom and room.id
```

This works for `useEffect`, `useLayoutEffect`, `useInsertionEffect`, `useMemo`,
`useCallback`, and `useImperativeHandle`. An explicit array keeps its exact React
meaning. Pass `null` for an intentional every-render effect or every-render
recomputation.

If the callback is created by opaque code, such as `useEffect(makeEffect())`, provide
an array or `null`. The compiler cannot inspect it without calling it again.

<h3>State hooks can read the latest value</h3>

`useState` and `useReducer` have an optional third tuple member: a stable function
that reads the latest scheduled state. It replaces the ref often used to avoid a
stale value after `await`.

```tsrx
const [draft, setDraft, getDraft] = useState('');

async function saveLater() {
	await waitForConnection();
	await save(getDraft());
}
```

`getDraft()` does not subscribe or render. During pending work it can be newer than
the DOM currently on screen. Ordinary two-item destructuring keeps the normal
two-item runtime path.

One small bailout detail is observable in render logging: after a real state change,
a same-value `setState` can skip Octane's component body entirely. React may enter
that body once more before it bails out. Children, effects, and the committed DOM are
unchanged.

<h2 id="events-and-dom">Events come from the browser</h2>

Octane delegates DOM events, but it does not wrap them in React's synthetic event
layer. Handlers receive the browser's real `Event` object. There is no event pooling,
and `event.currentTarget` is the element whose handler is running.

Bubbling, capture, `stopPropagation()`, portal bubbling, and native non-bubbling
events still reach the same logical ancestors as they do in React. The differences
are at the edges:

- Use `onInput` for each text edit. Native `change` fires when the browser commits
  the value, usually on blur.
- There are no synthetic `onChange`, `onBeforeInput`, or `onSelect` polyfills.
- Mouse and pointer enter/leave handlers use the browser's native per-element events;
  Octane does not synthesize them from over/out events.

For most controlled text fields, migration is one renamed prop:

```tsx
// React
<input value={text} onChange={(event) => setText(event.currentTarget.value)} />

// Octane
<input value={text} onInput={(event) => setText(event.currentTarget.value)} />
```

<h3>Controlled fields still stay controlled</h3>

`value` and `checked` work like React: the prop drives the DOM and is re-applied after
renders and discrete events. Radio groups, single and multiple selects, textareas,
and IME composition are covered. Use `defaultValue` or `defaultChecked` when the
browser should own the value.

Hydration preserves an edit made before Octane attaches, then the next real commit
re-applies a controlled value. A controlled text field without `onInput` gets a
development warning.

React 19 form actions and their hooks work too. One queueing edge differs: when an
action rejects, Octane continues later queued dispatches instead of cancelling them.

<h3>Attribute names stay close to HTML</h3>

Attribute values follow React's rules, including boolean attributes, overloaded
booleans, and common camelCase aliases. Native spellings such as `accept-charset`
also work directly. Octane has a small set of useful casing warnings rather than
React's exhaustive known-attribute table.

Unusual object values are converted with `toString()` and warned about in
development instead of throwing. On custom elements, a function-valued lowercase
`on*` prop attaches a listener; other props stay on the attribute path rather than
using React's property-or-attribute heuristic.

<h3>Classes compose without a helper</h3>

`class` and `className` accept strings, numbers, arrays, objects, and nesting.
Falsy entries drop out:

```tsrx
<article class={['card', isActive && 'active', { selected }]} />
```

React turns `className={['a', 'b']}` into `"a,b"`. Octane produces `"a b"` on the
client and server.

<h2 id="async-work">Transitions work without time slicing</h2>

Octane batches updates in a microtask, then runs each render to completion. It has
urgent and transition work, but no lanes, yield points, expiration, CPU time slicing,
or selective hydration.

That does not remove the useful part of transitions. A transition can keep the
current screen visible when its replacement suspends, and related boundaries can
reveal together. Priority controls those Suspense semantics; it does not interrupt a
render halfway through.

Two practical consequences:

- `flushSync` drains the whole update queue, including transition work.
- Passive effects always run after paint; `flushSync` does not pull them into the
  current call.

<h3>Independent `use()` calls start together</h3>

By default, the compiler removes avoidable Suspense waterfalls. It memoizes promise
creation by call site, starts provably independent `use()` inputs together, and
warms independent descendant fetches while a parent is suspended.

```tsrx
const user = use(fetchUser(id));
const teams = use(fetchTeams(id)); // starts with fetchUser, not after it
```

A real data dependency remains sequential, and unwrap order, error routing, and
hydration order stay React-compatible. Set `parallelUse: false` in the compiler or
Vite plugin when you specifically need React's serial waterfall timing.

<h3>External stores must notify</h3>

`useSyncExternalStore` works normally for stores that call their subscriber when
state changes. Unlike React, Octane does not perform an extra commit-time
`getSnapshot` read when the snapshot is unchanged and only the callback identity
changed. Its synchronous renderer does not have the concurrent gap that check
protects. A store that mutates silently between render and commit will therefore not
be discovered on that render.

<h2 id="errors-and-server">Errors and server rendering use Octane-shaped boundaries</h2>

<h3>Error boundaries are functions and templates</h3>

There are no class lifecycles for errors. Use `@try` / `@catch`, or the
`<ErrorBoundary>` component:

```tsrx
@try {
	<RiskyPanel />
} @catch (error, reset) {
	<button onClick={reset}>Try again</button>
};
```

Errors from rendering, teardown, and ref cleanup route to the nearest boundary.
An error with no boundary is reported through `console.error` rather than React's
`onUncaughtError` callback.

<h3>Buffered SSR returns styles beside the HTML</h3>

`renderToString`, `renderToStaticMarkup`, and `prerender` return
`{ html, css }`. The extra `css` field carries Octane's scoped styles.

The streaming APIs flush Suspense content out of order, but they are not Fizz
internals copied line for line. Octane re-runs the render in rounds instead of
incrementally rendering one boundary at a time, does not selectively hydrate, and
recreates head elements from a late streamed boundary on the client.

When hydration finds a value mismatch, Octane patches to the client value. A
structural mismatch is rebuilt in place with a warning instead of being thrown to a
boundary for a re-render. Function component output also carries Octane hydration
markers, so a server HTML snapshot is not necessarily identical to the same bare
host element.

<h2 id="reconciliation">Keyed lists move differently, not more often</h2>

Octane uses a longest-increasing-subsequence algorithm to minimize DOM moves. React
uses a different placement heuristic. The final order, the identity of surviving
nodes, focus, and component state are the same; only instrumentation that records
the exact nodes physically moved can see a difference.

<h2 id="not-supported">APIs Octane leaves out</h2>

Octane is built around function components, hooks, and the DOM. These React features
are intentionally absent:

- Class components, legacy roots, and class error-boundary lifecycles.
- Server Components, RSC/Flight, and `cache()`.
- `StrictMode` double-invocation, `Profiler`, and `SuspenseList`.
- `forwardRef` and `createRef`. Refs are ordinary props, including callback refs,
  object refs, and arrays of refs.
- The unstable CPU/commit-phase Suspense options and full React resource-hoisting
  machinery.
- Most `React.Children` utilities. Keyed `@for` is the normal collection API.

`useDebugValue` exists for library compatibility but has no visible runtime effect.
Resource hint functions—`preload`, `preinit`, `preconnect`, and `prefetchDNS`—are
supported.
