<h2 id="find-a-binding">Find the right binding</h2>

If you are moving an existing React app, start with the libraries it already uses. The
Octane package usually keeps the same concepts and API, with a different import:

```ts
// React
import { create } from 'zustand';

// Octane
import { create } from '@octanejs/zustand';
```

If you are starting a new app, choose by the problem you need to solve:

| What you need                     | Bindings                                                                                                                                         |
| --------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| Shared state                      | `@octanejs/zustand`, `@octanejs/jotai`, `@octanejs/redux`, `@octanejs/redux-toolkit`                                                             |
| Data fetching and caching         | `@octanejs/tanstack-query`                                                                                                                       |
| Routing                           | `@octanejs/tanstack-router`, `@octanejs/remix-router` (React Router-style APIs)                                                                  |
| UI and interaction                | `@octanejs/radix`, `@octanejs/base-ui`, `@octanejs/floating-ui`, `@octanejs/motion`, `@octanejs/dnd-kit`, `@octanejs/sonner`, `@octanejs/lucide` |
| Forms and rich-text editing       | `@octanejs/hook-form`, `@octanejs/lexical`                                                                                                       |
| Styling and localization          | `@octanejs/stylex`, `@octanejs/i18next`                                                                                                          |
| Tables, virtual lists, and charts | `@octanejs/tanstack-table`, `@octanejs/tanstack-virtual`, `@octanejs/recharts`                                                                   |
| Content and testing               | `@octanejs/mdx`, `@octanejs/testing-library`                                                                                                     |

Packages in the same row are not necessarily interchangeable. Use the upstream
library's documentation to learn its concepts, then check the Octane binding's supported
surface before relying on a particular API.

<h2 id="install-and-use">Install and use a binding</h2>

Install the `@octanejs` package alongside Octane. For example:

```sh
pnpm add @octanejs/zustand
```

Then import from the Octane package:

```tsrx
import { create } from '@octanejs/zustand';

type CounterStore = {
	count: number;
	increment: () => void;
};

const useCounter = create<CounterStore>(
	(set) => ({
		count: 0,
		increment: () => set((state) => ({ count: state.count + 1 })),
	}),
);

export function Counter() @{
	const count = useCounter((state) => state.count);
	const increment = useCounter((state) => state.increment);

	<button type="button" onClick={() => increment()}>
		{'Clicked ' + count + ' times'}
	</button>
}
```

The store API is still Zustand's; the binding makes its hook work with Octane. The same
idea applies to the other packages: learn the library from its upstream docs, then import
its Octane binding in your app.

<h2 id="check-support">Check support before shipping</h2>

Octane and its bindings are alpha software. Some bindings cover their upstream library
almost completely, while others intentionally support a smaller surface. Server rendering
and hydration coverage also varies.

The generated
[bindings status table](https://github.com/octanejs/octane/blob/main/docs/bindings-status.md)
is the source of truth for all 23 packages. It records:

- the upstream version being ported;
- which APIs are supported;
- known differences from the React package;
- server-rendering and hydration coverage;
- when that evidence was last reviewed.

A binding marked complete is complete for the version and surface described there. It is
not a blanket promise about every React-specific edge case.

<details className="deep-dive">
	<summary>How does a binding work?</summary>
	<div>
		<p>
			Many libraries split their framework-neutral logic from a thin React layer. Octane reuses the
			neutral core and replaces the React hooks and providers with Octane equivalents.
		</p>
		<p>
			The binding tests then run equivalent examples through Octane and the upstream React package.
			That is why most usage feels familiar even though a different renderer is doing the work.
		</p>
	</div>
</details>

<h2 id="app-tooling">App tooling</h2>

`@octanejs/vite-plugin` is app tooling rather than a library binding. It provides the
development server, routing, server rendering, hydration wiring, and production build used
by full Octane apps. The [quick start](/docs/quick-start) sets it up for you.
